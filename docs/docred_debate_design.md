# Re-DocRed 文档级关系融合：基于 PPDPP 思路的双智能体博弈设计

## 1. 总体思路

把每个文档中的每一对实体 `(h, t)` 看成一个独立博弈环境。

- **智能体A**：持有 LLM 对该实体对的候选关系列表 `C`。
- **智能体B**：持有 SLM 对该实体对的候选关系列表 `D`。
- **策略选择器 / Policy Selector**：观察当前状态后，决定下一步动作，目标是最大化全局 F1。

和 PPDPP 一样，这里把“最终预测”改写成：

1. `state`：当前实体对、候选关系、实体类型、A/B 是否一致、局部先验精度。
2. `action`：质疑、补充证据、接受、拒绝、结束。
3. `reward`：对最终三元组集合的精确率/召回率/F1 的贡献。

## 2. 实体对级别的状态定义

对于一个候选关系 `r`，状态可写为：

- 文档标题 `title`
- 实体对 `(h_idx, t_idx)`
- 头实体类型集合 `type(h)`
- 尾实体类型集合 `type(t)`
- `r ∈ C` 是否成立
- `r ∈ D` 是否成立
- 一致性状态：`both / slm_only / llm_only / none`
- SLM 的局部证据：分数、排序、与第二名的 margin
- 类型先验：`P(correct | relation, head_type, tail_type, source)`

本次实现中，策略选择器使用两个最稳健的状态特征：

- `P(correct | r, type(h), type(t), source=SLM)`
- `P(correct | r, type(h), type(t), source=LLM-only)`

## 3. 动作空间设计

推荐把动作空间设计成离散动作，便于直接套用 PPDPP 的策略网络范式。

### 核心动作

1. **KEEP_SLM**
   - 含义：接受智能体B(SLM)的当前关系。
   - 适用：`slm_only` 或 `both`，且 SLM 证据足够强。

2. **KEEP_INTERSECTION**
   - 含义：只保留 A/B 的共识关系。
   - 适用：A/B 给出同一关系，优先高精度保留。

3. **CHALLENGE_LLM**
   - 含义：要求智能体A为 `LLM-only` 关系辩护。
   - 适用：SLM 没给出该关系，但 LLM-only 在某些类型组合上历史精度不错，用于补召回。

4. **CHALLENGE_SLM**
   - 含义：要求智能体B为低质量 `SLM-only` 关系辩护。
   - 适用：SLM 给了关系，但类型先验较弱，可能是误报。

5. **REJECT**
   - 含义：拒绝当前关系。
   - 适用：A/B 都无法提供足够强的证据。

6. **STOP**
   - 含义：当前实体对的辩论结束，输出当前保留关系。

## 4. 提示词设计

### 给智能体A（LLM）的提示词

```text
你是智能体A，只能使用LLM候选集合C中的关系。
请针对当前关系给出：
1) 是否坚持；
2) 依据的实体类型约束；
3) 为什么它能补充SLM漏掉的召回。
```

### 给智能体B（SLM）的提示词

```text
你是智能体B，只能使用SLM候选集合D中的关系。
请针对当前关系给出：
1) 是否坚持；
2) 与实体类型是否匹配；
3) 为什么它比LLM更可靠。
```

### 给策略选择器的提示词

```text
你是策略选择器。观察当前状态(state)后，只能输出一个动作：
KEEP_SLM / KEEP_INTERSECTION / CHALLENGE_LLM / CHALLENGE_SLM / REJECT / STOP。
目标是在整个测试集上最大化F1。
```

## 5. 环境设计

### 5.1 一个 episode 的定义

一个 episode 对应一个实体对 `(h, t)`。

- 初始时环境给出 `C`、`D`、实体类型、文档标题和候选关系全集 `C ∪ D`。
- 选择器每次针对一个候选关系做动作决策。
- 所有候选关系处理完后执行 `STOP`，输出最终的关系集合。

### 5.2 状态转移

- `KEEP_SLM` / `KEEP_INTERSECTION` / `CHALLENGE_LLM`：把当前关系加入候选输出集合。
- `CHALLENGE_SLM` / `REJECT`：不加入输出集合。
- `STOP`：结束当前实体对 episode。

### 5.3 终止条件

- 当前实体对的候选关系全部处理完成；或
- 选择器显式输出 `STOP`。

## 6. 奖励函数设计

为了和 F1 直接对齐，建议使用**稀疏终局奖励 + 轻量过程奖励**。

### 6.1 终局奖励

对当前实体对输出的关系集合 `P(h,t)` 与真实集合 `G(h,t)` 计算：

- 真阳性 `TP = |P ∩ G|`
- 假阳性 `FP = |P - G|`
- 假阴性 `FN = |G - P|`

然后给出：

```text
R_final = 1.0 * TP - 1.0 * FP - 0.35 * FN
```

这样会更强地惩罚误报，同时给漏报一个较轻但明确的惩罚，符合本任务中“SLM 已经较强、LLM 主要用于补召回”的特点。

### 6.2 过程奖励

- 若 A/B 共识且被保留：`+0.10`
- 若低先验关系被拒绝：`+0.05`
- 若高先验 LLM-only 关系被正确补入：`+0.10`

### 6.3 全局目标

全局上仍以：

```text
maximize 2 * TP / (|Pred| + |Gold|)
```

作为策略选择器的最终训练目标。

## 7. 本仓库里的轻量实现

为了在当前代码库和运行环境下快速验证方案，我实现了一个**符号化策略选择器**：

- 对每个 `(relation, head_type, tail_type)` 估计历史精度。
- 如果是 **SLM** 候选，只有当类型组合精度高于阈值时才保留。
- 如果是 **LLM-only** 候选，只有当它在该类型组合上的历史精度高于阈值时才加入。

这相当于把“辩论后是否采纳”的策略学习，压缩成一个可解释、可复现、能直接优化 F1 的离散决策器。

## 8. 为什么这种设计比简单并集更好

简单并集 `LLM + SLM` 会明显拉低精确率，因为：

- SLM 已经有较高精度；
- LLM 的 `LLM-only` 预测总体精度较低；
- 但 LLM 在某些特定关系-类型组合上仍然有价值。

因此更合理的融合不是“全加”，而是：

- **默认信任 SLM，但过滤掉低类型先验误报；**
- **只在少数高质量类型组合上接纳 LLM-only 关系。**

这正好对应“两个智能体辩论 + 选择器判决”的机制。

## 9. 推荐训练流程

1. 用训练集/验证集统计 `(relation, head_type, tail_type, source)` 的精度先验。
2. 把这些特征输入策略选择器，动作空间使用上面的 6 个离散动作。
3. 用 REINFORCE / PPO 训练 selector。
4. 测试时按实体对逐个辩论，输出最终关系集合。

如果你后续想把它真正做成**可训练 RL 版本**，可以直接复用 PPDPP 当前的三件套：

- `Env`：把一个实体对封装成环境；
- `PPDPP`：把状态编码后输出动作概率；
- `reward`：用实体对级别的 TP/FP/FN 奖励监督策略更新。
